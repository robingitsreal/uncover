# Uncover — Design Spec

**Date:** 2026-04-15
**Owner:** Robin Ching
**Status:** Design approved, ready for implementation planning
**Purpose:** A portfolio-grade suite of AI agents for search engine optimization (SEO), generative engine optimization (GEO), and AI search optimization (AIO). Built to demonstrate hireable capability for content-strategy + GEO/AIO roles.

---

## 1. Goals and Non-Goals

### Goals
- Build a cohesive suite of **5 AI agents** covering SEO content strategy and GEO/AIO, branded as **Uncover**.
- Ship **3 artifacts** for hiring managers to evaluate: a public GitHub repo, a hosted live demo at `uncover.robinching.com`, and 5 published case studies on real brands.
- Demonstrate engineering taste across **agent framework choice**, **cost engineering**, **observability**, and **multi-tenant design**.
- Maintain a **hard monthly cost ceiling of $5** across all infrastructure and API spend.
- Design the `/connect` path for **bring-your-own Google account** from day one, so the tool is usable against a future employer's GSC/GA4 data on day one of hire.

### Non-Goals
- Not a production SaaS. No billing, no team features, no uptime SLA.
- Not a general-purpose SEO platform. The agents are focused tools, not an Ahrefs competitor.
- No consumer ChatGPT UI scraping. Cited as a known gap; not in scope.
- No custom-trained models. Off-the-shelf LLM APIs only.
- No open-ended public "try any URL" path for Performance Insights. That agent is case-study-only on the public surface.

---

## 2. Target Audience

Hiring managers for **content/SEO strategist** and **GEO/AIO specialist** roles. Secondary audience: AI engineers evaluating agent architecture.

The project must pass two simultaneous quality bars:
1. **SEO substance** — the outputs the agents produce must be recognizably useful to a working SEO practitioner.
2. **Engineering craft** — the code, architecture, and observability must hold up to scrutiny from an AI engineer.

### 3.2 Key architectural decisions

| Decision | Choice | Rationale |
|---|---|---|
| Split stack | Python agents + Next.js frontend | Matches real AI product shape; Python for framework signal, TS for demo polish |
| Python agent framework | **LangGraph** where stateful/parallel, **raw Anthropic SDK** where single-shot | Shows judgment: tool for the job, not framework-everything |
| Observability | **LangSmith** free tier for LangGraph agents; custom `run_steps` for SDK agents | LangSmith name recognition + unified DB-side logging |
| Frontend streaming | **SSE** from FastAPI to Next.js | Makes live runs feel alive; cheaper than WebSockets |
| Database | **Supabase Postgres** with RLS | Free tier, OAuth built-in, multi-tenant isolation |
| Background jobs | **GitHub Actions cron** | Free; nightly re-run of hero case study keeps dashboard fresh |
| Python hosting | **Modal** (pay-per-use, scales to zero) | Near-zero idle cost; cold starts acceptable for portfolio traffic |
| Frontend hosting | **Vercel Hobby** | Free; handles expected traffic |
| Domain | `uncover.robinching.com` | Existing domain, zero marginal cost |

### 3.3 Cost model

| Line item | Choice | Monthly cost |
|---|---|---|
| Domain | Subdomain on owned `robinching.com` | $0 |
| Frontend | Vercel Hobby | $0 |
| Python backend | Modal (scales to zero) | ~$0–1 |
| Database | Supabase Free | $0 |
| SERP + AI Overviews | **Serper.dev free tier** (2,500 queries/mo) | $0 |
| LLM calls — bulk extraction | **Gemini 2.5 Flash** free tier (1,500 req/day) | $0 |
| LLM calls — reasoning | **Claude Haiku** with prompt caching | ~$0–2 |
| LLM calls — synthesis (select spots) | **Claude Sonnet** with prompt caching | ~$0–2 |
| LangSmith traces | Free tier (5k/mo) | $0 |
| GitHub Actions cron | Free (2k min/mo) | $0 |
| **Target total** |  | **$0–5/month** |

**Hard kill-switch:** Anthropic org-level spend cap set to $5/mo. If tripped, agents return a graceful "budget exceeded" message rather than silently failing. Serper.dev's free tier is hard-capped; no overage possible.

**Query budget headroom:** ~590 SERP queries/month in realistic usage (150 cron + 40 bootstrap + 300 demo + 100 dev) vs. 2,500 free, leaving a 4× buffer.

### 3.4 Explicit swaps vs. the industry default

- **Serper.dev** instead of **SerpAPI** — SerpAPI's free tier (100/mo) cannot support a real portfolio demo. Serper returns equivalent SERP + AI Overview data. Called out in the README as deliberate cost engineering.
- **Modal** instead of **Fly.io / Railway** — Modal's scale-to-zero model fits portfolio-traffic shapes better. No idle-VM cost.
- **Gemini Flash** for bulk work instead of **Sonnet** — free tier + sufficient quality for parsing, entity extraction, and first-pass summarization.

## 4. The Agents

All five agents share: SSE status streaming, `runs`/`run_steps` logging, strict budget enforcement per run, and a consistent error/retry pattern.

### 4.1 GEO Citation Tracker (hero) — LangGraph

**Job:** Given a brand + list of tracked topics, measure where and how often the brand is cited by answer engines over time.

**Input:**
```json
{
  "brand_domain": "croovoo.com",
  "topics": ["dance formation software", "choreography app", ...]
}
```

**LangGraph shape:**
```
start
  → load_topics
  → fan_out (parallel per topic)
      ├─ perplexity_query    (Perplexity Sonar API)
      ├─ gemini_grounded     (Gemini 2.5 Flash + grounding, free tier)
      ├─ claude_web_search   (Anthropic web_search tool)
      ├─ openai_web_search   (OpenAI Responses API web_search tool)
      └─ google_ai_overview  (Serper.dev → AI Overview + organic results)
  → parse_citations          (extract cited URLs, detect brand, rank position)
  → persist                  (citation_observations table)
  → summarize_run            (daily delta: new/lost citations, SoV change)
```

**Why LangGraph earns its weight:** parallel per-engine fan-out with retries, partial-success handling, and LangSmith time-travel debugging when a run produces unexpected output. Hand-rolling this is roughly 300 lines of orchestration glue.

**Output:**
- Rows in `citation_observations(run_id, engine, topic, query, cited_url, is_target, position, snippet, ts)`
- A markdown run summary per execution

**Frontend view:**
- Line charts of share-of-voice per engine over time
- Sortable table of all observations
- Week-over-week diff view ("new citations gained", "citations lost")

**Nightly cron:** GitHub Actions fires daily at a fixed UTC hour, runs against the hero case study input.

### 4.2 Query Fan-Out Simulator — LangGraph

**Job:** For a target page, simulate the realistic tree of follow-up questions users ask answer engines, check whether the target surfaces, recommend content fixes where it doesn't.

**Flow:**
```
start
  → scrape_page (fetch rendered HTML, extract main content)
  → extract_entities (main entity + supporting claims)
  → generate_fanout_queries (~25 queries across intents:
      comparison, how-to, objection, adjacent-topic, alternatives)
  → run_queries (parallel per engine: Perplexity + Gemini grounded)
  → parse_citations (did the target surface?)
  → compute_surface_rate (percentage of queries where target appeared)
  → diagnose_failures (for non-surfacing queries: missing entity?
      missing schema? answer buried? wrong format?)
  → generate_remediation_list (ranked fixes)
```

**Why LangGraph:** conditional branching (diagnosis step only runs for non-surfacing queries), structured state across steps, retry logic.

**Output:**
- `fan_outs(run_id, target_url, query, engine, cited, diagnosis JSONB)`
- A ranked fix list in markdown

### 4.3 LLM-Readability Auditor — raw Anthropic SDK, tool-use

**Job:** Given a URL, score how well an LLM can extract and cite answers from the page.

**Tools exposed to Claude:**
- `fetch_rendered_html(url)` — JS-rendered fetch
- `extract_schema_org(html)` — parse JSON-LD
- `extract_headings(html)` — get heading tree
- `extract_entities(text)` — named entity extraction

**Flow:** Claude runs a short tool loop (typically 4–6 tool calls), then emits a structured report.

**Scorecard — 7 dimensions, 0–10 each:**
1. Answer-first writing (TL;DR early, lead with answers not setup)
2. Entity density (named entities per paragraph)
3. schema.org coverage (which types present, correctness)
4. Heading Q→A structure (headings phrased as questions with direct answers below)
5. Internal linking (relevance, density, anchor text quality)
6. `llms.txt` presence and quality
7. Crawlability (robots.txt, render-blocking, redirect chains)

**Output:** `audits(run_id, url, score, breakdown JSONB, fixes JSONB)`.

**Why raw SDK (not LangGraph):** the flow is linear tool use with no parallelism or branching. LangGraph would be overhead.

### 4.4 Content Brief Agent — raw Anthropic SDK, tool-use

**Job:** Keyword in → SEO *and* GEO-optimized content brief out.

**Tools exposed to Claude:**
- `serper_search(query)` — SERP + AI Overview
- `fetch_rendered_html(url)` — scrape a competitor
- `extract_entities(text)` — entity extraction for coverage analysis

**Flow:** Claude runs SERP, scrapes top 5 organic + reads AI Overview, extracts entity coverage across competitors, drafts a brief with:
- Target word count
- H2 outline
- Required entities
- Q&A block
- schema.org recommendations
- Citation references
- Internal-link anchor suggestions

**Output:** `briefs(run_id, keyword, target_domain, markdown, entities JSONB, citations JSONB)`.

### 4.5 Performance Insights Agent — LangGraph

**Job:** Pull GSC + GA4 + keyword expansion data, find patterns humans miss, output a prioritized action list.

**Inputs:** OAuth'd GSC property + GA4 property + optional target keyword list + date range (default: last 90 days vs. previous 90).

**Data sources (all free):**
- Google Search Console API — queries, pages, impressions, clicks, CTR, average position
- GA4 Data API — organic landing page sessions, engagement rate, conversions
- Serper.dev — "People Also Ask" + autocomplete for keyword expansion

**LangGraph shape:**
```
start
  → load_gsc
  → load_ga4
  → join_datasets (join on landing page URL)
  → pattern_agent (LLM with tools, adaptive):
      ├─ find_striking_distance(position 5–15, high impressions)
      ├─ find_low_ctr_outliers(CTR vs position benchmark)
      ├─ find_declining_pages(period-over-period)
      ├─ detect_cannibalization(same query, multiple URLs)
      └─ find_zero_conversion_pages(traffic but no GA4 events)
  → enrich_with_serper (People Also Ask for top opportunities)
  → prioritize_actions (LLM ranks by impact × effort)
  → generate_report (markdown + JSON for dashboard)
```

**Why LangGraph — and why this is the most "agent-shaped" of the suite:** the pattern agent chooses which analytical tools to run based on what the data shows, not a hardcoded pipeline. Different properties will take different paths.

**Output:** `insight_runs(run_id, gsc_property, ga4_property, date_range, findings JSONB, priorities JSONB, report_md)`.

**Example priority:** *"Page /formations/12-person is in striking distance: 42k impressions, position 9, 0.8% CTR. Rewrite title + add FAQ schema. Est. +2.5k clicks/mo."*

**Public access:** case-study-only at `/case-studies/croovoo-gsc-insights`. Anonymous visitors never see a connect button.

**Private access:** `/connect` path behind Supabase Auth → Google OAuth → property picker → live runs. Not linked from public navigation.

**Multi-tenancy:** RLS on `insight_runs` and `google_connections` — users only see their own data. Verified via a second test user before launch.

## 5. Data Model

### Core / shared

```sql
-- Supabase Auth handles users table

create table google_connections (
  id uuid primary key,
  user_id uuid references auth.users,
  google_email text,
  gsc_sites jsonb,         -- array of authorized GSC sites
  ga4_properties jsonb,    -- array of authorized GA4 properties
  created_at timestamptz default now()
);

create table runs (
  run_id uuid primary key,
  user_id uuid references auth.users,
  agent_type text,         -- 'citation_tracker' | 'fan_out' | 'readability' | 'brief' | 'insights'
  status text,             -- 'queued' | 'running' | 'succeeded' | 'failed' | 'budget_exceeded'
  started_at timestamptz,
  finished_at timestamptz,
  cost_usd numeric,        -- accumulated LLM + API cost
  input jsonb,
  error text
);

create table run_steps (
  id bigserial primary key,
  run_id uuid references runs,
  step_name text,
  status text,
  started_at timestamptz,
  finished_at timestamptz,
  payload jsonb            -- step inputs, outputs, tool calls
);
```

### Per-agent result tables

```sql
create table citation_observations (
  id bigserial primary key,
  run_id uuid references runs,
  engine text,             -- 'perplexity' | 'gemini' | 'claude' | 'openai' | 'google_aio'
  topic text,
  query text,
  cited_url text,
  is_target boolean,
  position int,
  snippet text,
  ts timestamptz default now()
);

create table fan_outs (
  id bigserial primary key,
  run_id uuid references runs,
  target_url text,
  query text,
  intent text,             -- 'comparison' | 'how_to' | 'objection' | 'adjacent' | 'alternative'
  engine text,
  cited boolean,
  diagnosis jsonb
);

create table audits (
  id bigserial primary key,
  run_id uuid references runs,
  url text,
  score int,               -- 0-70 total
  breakdown jsonb,         -- 7 dimension scores
  fixes jsonb
);

create table briefs (
  id bigserial primary key,
  run_id uuid references runs,
  keyword text,
  target_domain text,
  markdown text,
  entities jsonb,
  citations jsonb
);

create table insight_runs (
  id bigserial primary key,
  run_id uuid references runs,
  gsc_property text,
  ga4_property text,
  date_range tstzrange,
  findings jsonb,
  priorities jsonb,
  report_md text
);
```

### Curated / published

```sql
create table case_studies (
  slug text primary key,
  title text,
  agent_type text,
  target_brand text,
  run_id uuid references runs,
  summary_md text,
  published_at timestamptz
);
```

### RLS policies

- All result tables filter by `user_id` via `runs` join.
- `case_studies` is public-readable (anon role can SELECT).
- `google_connections` and `insight_runs` enforce strict user isolation.
- RLS verified by a second test user before launch (Definition of Done requirement).

## 7. Demo UX

### Public surface

- **Read-only case studies** carry most of the weight. Fast, polished, no API dependency.
- **"Try it" live runs** on 4 of 5 agents (Citation Tracker, Fan-Out, Readability Auditor, Content Brief). Visitor submits input, agent runs live, SSE streams status updates, result persists.
- Performance Insights has **no public try-it** — case-study-only.

### Abuse protection

- Rate limit: **3 runs per IP per day** across all agents.
- Hard monthly Anthropic budget cap at $5. If tripped, agents return "budget exceeded, try again next month" gracefully.
- Serper.dev free tier is hard-capped by the provider; no overage possible.
- Modal spend alerts configured.

### Private `/connect` path

- Supabase Auth → Google OAuth with `webmasters.readonly` + `analytics.readonly` scopes
- Property picker UI lists user's authorized GSC sites and GA4 properties
- Live runs of Performance Insights against the connected property
- RLS enforces per-user isolation
- Rate limit and budget cap enforced per-user (not per-IP)

## 12. Known Gaps and Honest Limitations

These are called out explicitly in the README. Acknowledging limits is itself a hiring signal.

- **Consumer ChatGPT UI is not polled.** There is no programmatic API and scraping the consumer surface is brittle and against ToS. The API-side `web_search` tool is polled instead.
- **Google AI Overviews come via Serper.dev, not direct scraping.** This is the accepted industry approach but introduces a third-party dependency.
- **Performance Insights has no public live demo.** OAuth-ing strangers into a portfolio app is more risk than signal. Case study only on the public surface.
- **Budget caps are hard.** If spend exceeds $5/month, public agents return graceful failures until next month. This is a feature, not a bug.
- **No custom model training or fine-tuning.** Off-the-shelf LLM APIs only.
